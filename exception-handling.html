基础 03：C++ 异常处理与错误边界

项目式机试不仅检查“正常输入能不能跑”，还会故意提供错误参数、非法状态和不存在的文件。一个可靠程序必须报告错误、保持状态一致，并在可恢复错误后继续处理下一条命令。

来自 25 年机试的直接要求：错误分为参数错误、状态错误和文件错误；错误信息以 Error: 开头；一条命令失败不应导致交互式程序终止。

1. 先建立错误处理模型

在一个交互式 CLI 中，错误处理可以分成三层：

层次职责例子
命令解析层检查参数数量、类型和值new -1 abc
业务操作层检查状态和文件，失败时提供具体原因没有活动位图却执行 save
命令循环边界统一捕获异常、输出 Error:，然后继续下一轮文件加载失败后仍能继续输入 new

异常适合把底层失败沿调用栈传递到统一边界。命令函数不必层层返回错误字符串，最外层也能集中保证输出格式一致。

2. throw、try 和 catch

#include <iostream>
#include <stdexcept>

double divide(double a, double b) {
    if (b == 0.0) {
        throw std::invalid_argument("divisor cannot be zero");
    }
    return a / b;
}

int main() {
    try {
        std::cout << divide(10, 0) << '\n';
    } catch (const std::invalid_argument& error) {
        std::cerr << "Error: " << error.what() << '\n';
    }
}
  1. throw 创建异常并立即停止当前正常流程。
  2. 运行时沿调用栈向外寻找匹配的 catch。
  3. 沿途已经构造完成的局部对象会自动析构，这个过程叫 stack unwinding。
  4. 找到处理器后执行 catch；如果始终没有匹配处理器，程序会终止。

3. 为什么要按 const& 捕获

catch (const std::exception& error) {
    std::cerr << error.what() << '\n';
}
  • 引用避免复制异常对象。
  • const 表示处理器只读取异常。
  • 基类引用能保留虚函数分派，what() 会返回实际派生异常的信息，避免 object slicing。
不要写 catch (std::exception error)。按值捕获会复制对象，并可能切掉派生类信息。

4. 异常的底层机制与自定义方法

4.1 异常类就是普通的类

异常对象和普通对象没有本质区别，只是用 throw 抛出。你可以在异常类中携带任何信息：

class MyError {
public:
    std::string message;
    int code;
    std::vector<int> line_numbers;  // 可以携带任何数据

    MyError(std::string msg, int c) : message(msg), code(c) {}
};

// 抛出
throw MyError(“something wrong”, 42);

// 捕获后可以访问所有字段
try {
    // ...
} catch (const MyError& e) {
    std::cout << e.message << “ (code: “ << e.code << “)\n”;
}

4.2 标准异常类的继承体系

C++ 标准库的异常都继承自 std::exception：

std::exception (基类，有 what() 方法)
├── std::logic_error (程序逻辑问题)
│   ├── std::invalid_argument (参数非法)
│   ├── std::out_of_range (越界)
│   └── std::length_error (长度错误)
├── std::runtime_error (运行时环境问题)
│   ├── std::range_error (范围错误)
│   ├── std::overflow_error (溢出)
│   └── std::underflow_error (下溢)
└── std::bad_alloc (内存分配失败)

4.3 为什么要继承 std::exception？

继承的好处是统一接口。所有异常都有 what() 方法，可以用基类捕获：

class MyError : public std::exception {
private:
    std::string msg;

public:
    MyError(const std::string& message) : msg(message) {}

    // 必须实现这个方法
    const char* what() const noexcept override {
        return msg.c_str();
    }
};

// 可以用基类捕获所有异常
catch (const std::exception& e) {
    std::cerr << e.what() << “\n”;  // 无论什么异常，都能调用 what()
}

4.4 最实用的方式：继承 std::runtime_error

std::runtime_error 已经帮你处理了字符串存储和 what() 实现，你只需要添加自己的字段：

#include <stdexcept>

class CommandError : public std::runtime_error {
public:
    int error_code;  // 自定义字段

    CommandError(const std::string& msg, int code = 0)
        : std::runtime_error(msg), error_code(code) {}
};

// 使用
throw CommandError(“Invalid arguments”, 1);

// 捕获
catch (const CommandError& e) {
    std::cerr << e.what() << “\n”;       // 自动有 what()
    std::cerr << e.error_code << “\n”;  // 访问自定义字段
}

4.5 机考最简方案（推荐）

如果不需要额外字段，只需要区分异常类型，可以这样写：

class CommandError : public std::runtime_error {
public:
    using std::runtime_error::runtime_error;  // 继承构造函数
};

// 这一行等价于手写：
// CommandError(const std::string& msg) : std::runtime_error(msg) {}

为什么这样做？

  • 不需要写构造函数
  • 自动有 what() 方法
  • 可以通过消息内容区分错误
  • 和 STL 内部异常（如 std::bad_alloc）区分开

4.6 完整的自定义异常示例

如果需要携带更多信息（如文件路径、坐标、错误类型枚举），可以这样写：

#include <stdexcept>
#include <string>

class BitmapError : public std::runtime_error {
public:
    enum class Type {
        NO_ACTIVE,
        INVALID_SIZE,
        FILE_ERROR
    };

    Type type;
    std::string filepath;  // 可选字段
    int width, height;     // 可选字段

    // 构造函数 1：一般错误
    BitmapError(Type t, const std::string& msg)
        : std::runtime_error(msg), type(t), width(0), height(0) {}

    // 构造函数 2：文件错误
    BitmapError(const std::string& msg, const std::string& path)
        : std::runtime_error(msg), type(Type::FILE_ERROR),
          filepath(path), width(0), height(0) {}

    // 构造函数 3：尺寸错误
    BitmapError(const std::string& msg, int w, int h)
        : std::runtime_error(msg), type(Type::INVALID_SIZE),
          width(w), height(h) {}
};

// 使用
throw BitmapError(BitmapError::Type::NO_ACTIVE, “No active bitmap”);
throw BitmapError(“Cannot open file”, “/tmp/img.txt”);
throw BitmapError(“Size too large”, 5000, 3000);

// 捕获后可以根据类型做不同处理
catch (const BitmapError& e) {
    std::cerr << “Error: “ << e.what() << “\n”;
    if (e.type == BitmapError::Type::FILE_ERROR) {
        std::cerr << “Affected file: “ << e.filepath << “\n”;
    } else if (e.type == BitmapError::Type::INVALID_SIZE) {
        std::cerr << “Invalid size: “ << e.width << “x” << e.height << “\n”;
    }
}
机考建议：对于大部分题目，用最简方案（只继承 std::runtime_error，不添加字段）就够了。只有当需要根据错误类型做不同处理时，才需要添加枚举或额外字段。

5. 标准异常类型速查（了解即可）

如果你想使用标准库提供的异常类型，这里是常用的几种：

类型适合场景示例
std::invalid_argument函数参数值不合法throw std::invalid_argument(“width must be positive”);
std::out_of_range索引、坐标超出范围throw std::out_of_range(“index out of bounds”);
std::runtime_error运行时环境问题throw std::runtime_error(“file not found”);

但在机考中，用一个自定义类型通常更简单，不需要记住这些标准类型的区别。

6. 捕获位置决定程序能否继续

交互式工具必须把 try/catch 放在命令循环内部。这样一轮失败只会结束当前命令：

std::string line;
while (std::cout << "> " << std::flush,
       std::getline(std::cin, line)) {
    try {
        execute_command(line);
    } catch (const CommandError& error) {
        std::cout << "Error: " << error.what() << '\n';
    }
}

如果把 try/catch 放在整个 while 外面，第一次异常会直接跳出循环；即使随后捕获，程序也不会再读取下一条命令。这正好违反 25 年机试的要求。

6. 三类错误如何抛出

参数错误

void do_new(std::istringstream& args) {
    int width, height;
    std::string extra;
    if (!(args >> width >> height) ||
        (args >> extra) ||
        width <= 0 || height <= 0 ||
        width > 2000 || height > 2000) {
        throw CommandError("Invalid arguments.");
    }
    // 参数完全有效后再创建位图。
}

状态错误

void require_active_bitmap(const std::optional<Bitmap>& active) {
    if (!active) {
        throw CommandError("No active bitmap.");
    }
}

文件错误

std::ifstream input(path);
if (!input) {
    throw CommandError("Cannot open file '" + path + "'.");
}

错误消息应该告诉用户失败对象和原因，但不要把“打不开文件”一律误报成“格式错误”。打开失败与内容校验失败是两个不同阶段。

7. 失败时不能破坏已有状态

假设已经有一张活动位图，用户又执行 load broken.txt。正确行为是报告错误，并继续保留原位图。不要一开始就清空 active。

void do_load(std::optional<Bitmap>& active,
             const std::string& path) {
    Bitmap next = load_and_validate(path); // 可能抛异常
    active = std::move(next);               // 全部成功后才提交
    std::cout << “Bitmap '” << path << “' loaded.\n”;
}

这里遵循”先在临时对象中完成读取与校验，最后一次性替换状态”的原则。失败发生在赋值之前，因此原状态不变。这是项目题里非常重要的 strong exception guarantee。

7.1 std::move 是什么？

std::move 的作用是把左值转换成右值引用，告诉编译器”这个对象我不再需要了，你可以移动它的资源而不是复制”。

std::vector<int> a = {1, 2, 3, 4, 5};
std::vector<int> b = a;              // 复制：a 和 b 各有一份数据
std::vector<int> c = std::move(a);   // 移动：c 拿走 a 的数据，a 变成空的

7.2 为什么要用 std::move？

  • 避免不必要的复制：对于大对象（如 vector、string），复制开销很大
  • 转移资源所有权：明确表示”我不再需要这个对象”

7.3 常见使用场景

场景 1：返回大对象时

Bitmap load_file(const std::string& path) {
    std::vector<std::string> rows;
    // 读取文件填充 rows...
    return Bitmap{std::move(rows)};  // 移动而不是复制
}

场景 2：替换状态时

void do_load(std::optional<Bitmap>& active, const std::string& path) {
    Bitmap next = load_file(path);
    active = std::move(next);  // 移动到 active，避免复制大对象
}

场景 3：放入容器时

std::vector<std::string> lines;
std::string line = “very long string...”;
lines.push_back(std::move(line));  // 移动而不是复制
// 此时 line 已经被掏空，不能再使用

7.4 注意事项

  • std::move 之后，原对象处于”有效但未指定”的状态
  • 通常原对象会变成空的（如空 vector、空 string）
  • 不要在 move 之后再使用原对象（除非重新赋值）
std::string s1 = “hello”;
std::string s2 = std::move(s1);
std::cout << s1;  // 错误！s1 已经被掏空，不应再使用
s1 = “world”;     // 正确：重新赋值后可以使用
机考建议：在返回大对象、替换状态、放入容器时使用 std::move。记住规则：move 之后原对象就”废”了，不要再用。

8. RAII：异常发生后资源为什么不会泄漏

C++ 的资源安全依赖 RAII：资源由对象拥有，对象析构时释放资源。异常展开调用栈时，局部对象仍会正常析构。

Bitmap load_and_validate(const std::string& path) {
    std::ifstream input(path); // 析构时自动关闭文件
    if (!input) {
        throw CommandError("Cannot open file '" + path + "'.");
    }

    std::vector<std::string> rows; // 自动管理内存
    // 读取或校验失败时直接 throw；不需要手动 close/free。
    return Bitmap{std::move(rows)};
}
实践原则：优先使用 std::vector、std::string、文件流和智能指针；不要在拥有裸资源后依赖手写清理代码。

9. 哪些情况不应该使用异常

  • 正常控制流不要用异常，例如循环结束、用户输入 exit。
  • 题目明确规定“非法绘制范围安全忽略”时，这是一条业务规则，不应该抛异常。
  • 高频内层循环中的普通判断不要改成异常，例如像素裁剪和 BFS 邻居越界。
  • assert 用于程序员不变量，不用于可预期的用户输入错误。

异常适合“当前操作无法按契约完成，正常路径必须立刻中断”的情况；普通分支适合预期内、频繁发生的业务选择。

10. 异常、错误消息与退出码的关系

机制边界作用
异常程序内部把失败从业务函数传到统一处理位置
Error: 消息程序与用户解释当前命令为什么失败
退出码程序与调用者/脚本说明整个进程最终是否成功

25 年机试是交互式工具：某条命令失败不等于整个进程失败，因此捕获异常后继续循环；真正无法启动或发生不可恢复的顶层错误时，才结束程序并返回非零退出码。

11. 完整示例：不会因错误退出的命令循环

#include <fstream>
#include <iostream>
#include <optional>
#include <sstream>
#include <stdexcept>
#include <string>
#include <utility>
#include <vector>

class CommandError : public std::runtime_error {
public:
    using std::runtime_error::runtime_error;
};

struct Bitmap {
    std::vector<std::string> rows;
};

Bitmap load_and_validate(const std::string& path) {
    std::ifstream input(path);
    if (!input) {
        throw CommandError("Cannot open file '" + path + "'.");
    }

    std::vector<std::string> rows;
    std::string row;
    while (std::getline(input, row)) {
        rows.push_back(row);
    }
    while (!rows.empty() && rows.back().empty()) {
        rows.pop_back();
    }
    if (rows.empty()) {
        throw CommandError("Invalid file format.");
    }

    const std::size_t width = rows.front().size();
    for (const std::string& current : rows) {
        if (current.size() != width ||
            current.find_first_not_of(".#") != std::string::npos) {
            throw CommandError("Invalid file format.");
        }
    }
    return Bitmap{std::move(rows)};
}

void require_no_extra(std::istringstream& args) {
    std::string extra;
    if (args >> extra) {
        throw CommandError("Invalid arguments.");
    }
}

int main() {
    std::optional<Bitmap> active;
    std::string line;

    while (std::cout << "> " << std::flush,
           std::getline(std::cin, line)) {
        std::istringstream args(line);
        std::string command;
        if (!(args >> command)) {
            continue;
        }

        try {
            if (command == "load") {
                std::string path;
                if (!(args >> path)) {
                    throw CommandError("Invalid arguments.");
                }
                require_no_extra(args);
                Bitmap next = load_and_validate(path);
                active = std::move(next);
                std::cout << "Bitmap '" << path << "' loaded.\n";
            } else if (command == "info") {
                require_no_extra(args);
                if (!active) {
                    throw CommandError("No active bitmap.");
                }
                std::cout << "Width: " << active->rows.front().size()
                          << "\nHeight: " << active->rows.size() << '\n';
            } else if (command == "exit") {
                require_no_extra(args);
                break;
            } else {
                throw CommandError("Unknown command.");
            }
        } catch (const CommandError& error) {
            std::cout << "Error: " << error.what() << '\n';
        } catch (const std::exception& error) {
            std::cout << "Error: Internal failure: "
                      << error.what() << '\n';
        }
    }
    return 0;
}

printf 'info\nload missing.txt\nbad\nexit\n' | ./exception_cli

12. 常见错误

  1. 把 try/catch 放在命令循环外，导致第一次错误就结束交互。
  2. 使用 catch (...) 后什么都不输出，悄悄吞掉未知错误。
  3. 按值捕获异常，造成复制和 object slicing。
  4. 先修改活动状态，再进行可能失败的文件读取或校验。
  5. 在析构函数中抛异常；栈展开期间再次抛出可能直接触发 std::terminate。
  6. 把用户输入错误写成 assert，Release 构建中断言可能被关闭。
  7. 所有异常都只输出 Invalid arguments，丢失文件路径和具体失败原因。

13. 机试实现顺序

  1. 先确定统一错误类型和 Error: 输出边界。
  2. 先写命令循环，并验证异常后可以继续读取下一条命令。
  3. 把参数校验、状态校验和文件校验分别放在最接近数据的位置。
  4. 修改共享状态前，先在临时对象中完成所有可能失败的工作。
  5. 最后补错误路径测试；不要只测试成功样例。
掌握标准：能让业务函数用异常报告失败；能在命令循环内部统一捕获；能保证一条命令失败后程序继续运行，且原有活动状态不被破坏。
