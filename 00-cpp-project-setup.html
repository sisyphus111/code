00 从零构建 C++ 项目

这一节只解决两种最常见的构建场景：单文件程序直接用 g++，多文件项目用 CMake。

选择方法：只有一个 main.cpp 时不需要 CMake；出现多个 .cpp、头文件或第三方库后，再用 CMake 统一管理构建规则。

1. 单文件：直接使用 g++

最小程序只需要一个入口函数 main：

#include <iostream>

int main() {
    std::cout << "Hello, C++!\n";
    return 0;
}

将代码保存为 main.cpp，用下面两条命令编译并运行：

g++ -std=c++17 -Wall -Wextra -O2 main.cpp -o main
./main
参数含义
-std=c++17使用 C++17 标准；机试前应确认允许的标准版本
-Wall -Wextra打开常用警告，尽早发现可疑代码
-O2开启常用优化；调试阶段也可以先用 -O0 -g
main.cpp需要编译的源文件
-o main把生成的可执行文件命名为 main

标准输入、标准输出与重定向

算法题通常从标准输入读取，向标准输出写结果：

#include <iostream>

int main() {
    int a, b;
    std::cin >> a >> b;
    std::cout << a + b << '\n';
}
echo "3 5" | ./main          # 把字符串作为程序输入
./main < input.txt           # 从文件读取输入
./main < input.txt > out.txt # 输入和输出都重定向
diff out.txt answer.txt       # 比较实际输出与标准答案

调试版本与提交版本

# 调试：保留调试信息，并开启运行时错误检查
g++ -std=c++17 -Wall -Wextra -O0 -g \
    -fsanitize=address,undefined main.cpp -o main

# 提交前：使用常见优化参数
g++ -std=c++17 -Wall -Wextra -O2 main.cpp -o main

AddressSanitizer 可以发现数组越界、use-after-free 等内存错误；UndefinedBehaviorSanitizer 可以发现有符号整数溢出等未定义行为。部分机试环境不提供 sanitizer，因此它们主要用于本地调试，不要把 sanitizer 参数带进最终提交环境。

2. 从源码到可执行文件

  1. 预处理：展开 #include、宏和条件编译。
  2. 编译：检查 C++ 语法与类型，把每个源文件变成目标文件。
  3. 链接：把目标文件和所需库连接成最终可执行文件。

因此错误也分两类：看到 syntax error、类型不匹配通常是编译错误；看到 undefined reference 或 Undefined symbols 通常是链接阶段找不到某个函数的实现。

3. 多文件项目的结构

常见结构是头文件声明接口，源文件提供实现，main.cpp 负责组织程序流程：

hello_cpp/
├── CMakeLists.txt
├── include/
│   └── calculator.h
└── src/
    ├── calculator.cpp
    └── main.cpp

include/calculator.h：

#pragma once

int add(int a, int b);

src/calculator.cpp：

#include "calculator.h"

int add(int a, int b) {
    return a + b;
}

src/main.cpp：

#include <iostream>
#include "calculator.h"

int main() {
    std::cout << add(3, 5) << '\n';
}

calculator.h 声明模块向外提供的接口，calculator.cpp 实现接口，main.cpp 只负责程序入口和整体流程。接下来由 CMake 统一管理这些文件。

4. 用 CMake 构建多文件项目

CMake 不是编译器。它读取 CMakeLists.txt，生成适合当前平台的构建系统，再由该构建系统调用编译器。

项目根目录的 CMakeLists.txt：

cmake_minimum_required(VERSION 3.16)
project(hello_cpp LANGUAGES CXX)

set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)
set(CMAKE_CXX_EXTENSIONS OFF)

add_executable(calculator
    src/main.cpp
    src/calculator.cpp
)

target_include_directories(calculator PRIVATE include)
target_compile_options(calculator PRIVATE -Wall -Wextra)
CMake 命令作用
project(...)声明项目名称及使用的语言
add_executable(...)创建一个可执行目标，并列出组成它的源文件
target_include_directories(...)为指定目标添加头文件搜索目录
target_compile_options(...)为指定目标添加编译选项

问：为什么 add_executable 里没有列出 .h？

因为 .cpp 和 .h 在构建中的角色不同：

  • .cpp 是编译单元。每个 .cpp 都会被单独编译成目标文件，最后再链接，因此必须让构建系统知道哪些 .cpp 属于这个目标。
  • .h 通常不是单独编译的。某个 .cpp 写下 #include "calculator.h" 后，预处理器会把头文件内容纳入这个 .cpp 的编译过程。

头文件的位置由下面这行标明：

target_include_directories(calculator PRIVATE include)

它告诉编译器：编译目标 calculator 时，把项目的 include/ 加入头文件搜索路径。因此源码中的：

#include "calculator.h"

可以找到 include/calculator.h。这大致等价于直接使用 g++ 时添加：

-Iinclude
关系总结：add_executable 回答“编译并链接哪些实现文件”，target_include_directories 回答“去哪些目录查找头文件”，而源码中的 #include 回答“当前编译单元实际需要哪个头文件”。

可以把头文件也写进 add_executable 吗？

可以，例如：

add_executable(calculator
    src/main.cpp
    src/calculator.cpp
    include/calculator.h
)

这样做主要是让 IDE 更完整地展示目标包含的文件，并不意味着头文件会被单独编译。对于这个简单项目，不列出头文件也能正确构建。

标准的三步工作流

# 1. 配置：读取当前目录，生成构建规则到 build/
cmake -S . -B build -DCMAKE_BUILD_TYPE=Debug

# 2. 构建：让 CMake 调用底层构建工具和编译器
cmake --build build

# 3. 运行
./build/calculator

-S . 指源码根目录，-B build 指构建产物目录。把产物放进独立的 build/ 叫作 out-of-source build，可以避免目标文件和临时文件污染源码目录。

问：cmake --build build 后，可执行文件在 build/ 里吗？

通常是。如果目标定义为 add_executable(calculator ...)，使用 Unix Makefiles 或 Ninja 等单配置生成器时，常见路径是：

build/calculator

因此可以直接运行：

./build/calculator

但“在 build/ 里”不一定表示“紧挨着 build/ 根目录”，还有两种常见情况：

  • Visual Studio、Xcode 等多配置生成器通常增加配置子目录，例如 build/Debug/calculator 或 build/Release/calculator。
  • 项目设置了 RUNTIME_OUTPUT_DIRECTORY 时，可执行文件会生成到项目指定的位置。

不确定具体位置时，可以让 CMake 显示详细构建命令，或在构建目录中查找：

cmake --build build --verbose
find build -type f -perm -111

修改代码后需要执行哪一步

  • 只修改了 .cpp 或 .h：通常重新执行 cmake --build build。
  • 修改了 CMakeLists.txt：构建时通常会自动重新配置；不确定时再运行一次配置命令。
  • 切换编译器、生成器或重要配置：删除旧 build/ 后重新配置，避免缓存冲突。

5. Debug 与 Release

模式特点适合场景
Debug包含调试信息、优化较少，便于定位代码开发、断点调试、排错
Release开启优化，运行通常更快，但调试体验较差性能测试、最终提交前验证
 
cmake -S . -B build-debug -DCMAKE_BUILD_TYPE=Debug
cmake --build build-debug

cmake -S . -B build-release -DCMAKE_BUILD_TYPE=Release
cmake --build build-release

6. 最常见的错误怎么判断

现象先检查什么
command not found: cmakeCMake 是否安装，以及可执行文件是否位于 PATH
fatal error: xxx.h: No such file头文件名是否正确，include 目录是否通过 -I 或 CMake 加入
undefined reference函数是否只有声明没有实现，或实现所在的 .cpp 是否加入构建目标
./program: no such file编译是否成功、程序实际生成在哪个目录、文件名是否正确
修改后运行结果没变化是否重新构建，以及运行的是不是另一个目录中的旧程序
排错原则：先看终端输出的第一条错误，而不是最后一条；后面的错误经常只是第一处失败引发的连锁反应。

7. 从零开始时的固定动作

  1. 只有一个 main.cpp 时，先用 g++ 直接编译运行，不引入 CMake。
  2. 先让最小程序成功运行，再开始添加算法或业务逻辑。
  3. 需要拆分模块时，建立 src/、include/ 和根目录的 CMakeLists.txt。
  4. 用 cmake -S . -B build 配置，用 cmake --build build 构建。
  5. 每增加一个源文件，都把它加入对应的 add_executable 或 add_library。
  6. 提交前使用多组输入测试，并用 Release 模式检查性能。

8. 一分钟速查

# 单文件：g++
g++ -std=c++17 -Wall -Wextra -O2 main.cpp -o main
./main < input.txt

# 多文件：CMake
cmake -S . -B build -DCMAKE_BUILD_TYPE=Debug
cmake --build build
./build/程序名

掌握标准：能独立用 g++ 构建单文件程序；能根据 CMakeLists.txt 构建多文件项目；看到编译或链接错误时知道从哪一层检查。
